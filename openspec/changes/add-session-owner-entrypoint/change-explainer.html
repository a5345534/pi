Session Owner Entry Point

Why：為什麼需要這個邊界

@a5345534/pi-session 已經從 @a5345534/pi-coding-agent 機械抽取出來，但目前仍主要是 session persistence utilities，而不是通用 session owner entrypoint。live-session / Web attach 的後續方向需要先確定：誰是 live runtime owner、誰能寫 JSONL、前端或 bridge 只能透過什麼介面互動。

若沒有這個邊界，後續很容易出現兩種失敗模式：第一，多個 frontend 各自複製 runtime responsibility；第二，bridge 只寫 JSONL，卻沒有擁有 streaming、pending prompts、tool calls、abort、compaction、branch selection 與 event ordering。

決策摘要：packages/session 走向 app-neutral session owner/client entrypoint；packages/coding-agent 保留 Pi 應用層服務與 pi-fork defaults；live session JSONL 必須由擁有 live runtime 的唯一 owner 寫入；pi-sessiond、Web attach、network protocol 與 remote auth 均是後續另案。

What Changes：這次審查要批准什麼

要建立的方向

  • 把 packages/session 定義為 app-neutral session persistence 與 future owner/client contract 所在地。
  • 把 packages/coding-agent 定義為 Pi application host，負責 CLI/TUI/RPC/SDK、settings、auth、models、tools、resources、extensions 與 pi-fork config defaults。
  • 採 staged migration：先移除 session package 的 app-specific config，再加 owner/client types 與 contract tests，再由 coding-agent 提供 current runtime adapter。
  • 建立 single-writer 原則：live session JSONL writer 必須也是 runtime owner。

不在這次交付

  • 不建 pi-sessiond。
  • 不建 Web attach mode、browser UI 或 remote protocol。
  • 不加入 multi-user roles、permissions、network authentication 或 remote-host discovery。
  • 不改 JSONL schema、不搬 provider/tool/resource/extension/TUI/CLI 到 packages/session。

Before / After：變更前與變更後

變更前

  • packages/session 是 helper/persistence package，仍含 pi-fork app-specific defaults。
  • AgentSession 與 AgentSessionRuntime 在 packages/coding-agent 內擁有 live runtime。
  • TUI/RPC/SDK 等模式直接靠 coding-agent runtime API 協作。
  • 尚無可讓未來 Web 或 daemon 共用的 app-neutral owner/client contract。

變更後

  • packages/session 不再自行決定 pi-fork path/config，改由 host 傳入 storage options。
  • packages/session 定義 SessionOwner、SessionClient、commands、events、snapshot contracts。
  • packages/coding-agent 先以 in-process adapter 包住既有 AgentSessionRuntime。
  • 後續 TUI/Web/RPC/SDK 可以透過同一 owner boundary attach，而不是各自寫 JSONL。

Decision Points：需要審查的決策點

D1. packages/session owns app-neutral session boundary

Choicesession package 負責 persistence helpers、session identity/path、tree/message contracts、owner/client contracts、commands/events/snapshots、single-writer semantics。

Trade-off先建立穩定 boundary，而不是立即搬完整 runtime；速度較慢，但風險較低。

Recommendation批准。這是 Web/TUI parity 的必要前置條件。

D2. coding-agent remains application host

ChoiceCLI、TUI、RPC、SDK、settings、auth、models、resources、extensions、tools、pi-fork defaults 留在 coding-agent。

Trade-offsession package 需要 dependency injection seam；但避免共享 package 被 app-specific concern 汙染。

Recommendation批准。這是 module boundary 的核心。

D3. Runtime ownership migrates through adapter first

Choice先由 coding-agent 提供 in-process SessionOwner adapter 包住現有 AgentSessionRuntime。

Trade-off短期有 adapter layer；長期降低大搬 runtime 的破壞面。

Recommendation批准，並要求 contract tests 覆蓋後再搬實作。

D4. Live JSONL writer must own runtime

High riskbridge/client 不得直接 append live JSONL，除非它也擁有 streaming、queue、tool calls、abort、compaction、branch state 與 ordering。

Trade-off禁止快速但不完整的 JSONL bridge；換取資料與 runtime state 一致性。

Recommendation強烈批准。這是避免 corruption 與 duplicated owner 的關鍵。

D6. packages/session exposes types-only in first phase

Choicepackages/session 定義 contract types (SessionOwner, SessionClient, SessionCommand, SessionOwnerEvent, SessionSnapshot) 與 contract tests，但不提供 in-process reference owner。

Trade-off沒有 reference owner 讓測試稍微間接，但避免 session package 需要 model invocation、tool execution 等應屬於 coding-agent 的 runtime 依賴。

Recommendation批准。contract tests 已足以驗證 types 可實作；coding-agent adapter (D3) 是第一個真正的實作。

D7. Event surface classification: public vs internal

ChoiceSessionOwnerEvent 的 stable public names 為 session.created, session.restored, session.closed, snapshot, turn.started, turn.completed, message.started, message.delta, message.completed, tool.started, tool.progress, tool.completed, compaction.started, compaction.completed, model.changed, thinking.changed, error。

Internal detailsJSONL entry types (SessionHeader, SessionMessageEntry, CompactionEntry 等)、coding-agent extension events (ProjectTrustEvent, ResourcesDiscoverEvent 等) 與 adapter mapping 均為內部細節，非 public contract。

Recommendation批准。避免 public contract 被 JSONL schema 變更或 extension API 演進所耦合。

Scope Boundaries：開發與不開發

本 change 要開發

  • host-provided session storage/path options。
  • SessionOwner / SessionClient / commands / events / snapshots types。
  • single-writer rule 的 normative spec 與 tests。
  • coding-agent in-process adapter over AgentSessionRuntime。
  • 低風險 call sites 逐步走 owner/client contract。

明確不開發

  • pi-sessiond daemon。
  • Web attach mode、browser UI、WebSocket/stdio/IPC wire protocol。
  • remote auth、role permission、multi-user policy、cross-host discovery。
  • JSONL schema migration。
  • 把 provider auth、model registry、tool definitions、resources、extensions、TUI、CLI 搬進 packages/session。

Implementation Slices：實作切片

Slice 0：Session package 去 app-specific

Acceptance criteria：packages/session 不再 hardcode pi-fork、.pi-fork、PI_FORK_*；由 coding-agent 傳入 effective paths。

Dependency：現有 SessionManager 行為與 JSONL compatibility。

Rollback：保留 coding-agent shim 與原有 path computation，必要時只回退 path injection seam。

Slice 1：Owner/client contract

Acceptance criteria：新增 app-neutral types 與 contract tests，覆蓋 command serialization、event subscription、snapshot retrieval、single-writer expectation。

Dependency：公開 terms 確認：SessionOwner、SessionClient、SessionCommand、SessionOwnerEvent、SessionSnapshot。

Rollback：types 可保持 experimental，不影響現有 runtime path。

Slice 2：Coding-agent adapter

Acceptance criteria：coding-agent 提供 in-process owner adapter，映射 create/open/new/resume/fork/import、prompt、abort、compact、shutdown 等 supported operations。

Dependency：現有 AgentSessionRuntime 與 AgentSession lifecycle。

Rollback：frontend call sites 可暫時維持 direct runtime calls。

Slice 3：Call-site migration

Acceptance criteria：低風險 SDK/RPC call sites 先改用 client contract；interactive mode 需等 tests 覆蓋 session switching、forking、queueing、abort、tool calls、compaction、JSONL ordering。

Dependency：adapter event mapping 不暴露 TUI/provider/tool implementation objects。

Rollback：保留 compatibility shims 到 replacement imports 有文件與測試。

Verification Plan：驗證計畫

  • 實作後執行 npm run check。
  • 執行 npm --prefix packages/session test，包含新增 contract tests。
  • 針對改動 call sites 執行 packages/coding-agent session switching、forking、RPC、SDK、extension lifecycle tests。
  • runtime 行為改動後手動 smoke：pi-fork --help、pi-fork --version、以及 no-session/no-tools prompt。
  • 檢查 packages/session 沒有新增對 packages/coding-agent、TUI、app config、provider auth、CLI parsing 的 imports。
  • 驗證 live session 只有 active owner 寫 JSONL，frontend bridge/client 不直接寫入。

Risk Register：風險清單

High riskBoundary 過寬，session 變成第二個 coding-agent

Mitigation：明確排除 provider/tool/resource/extension/TUI/CLI，所有 app-specific services 由 coding-agent injection。

High riskDirect JSONL writer 回流

Mitigation：spec 規定 live JSONL writer 必須是 runtime owner；bridge/client 只能送 command 給 owner。

MediumOwner events 洩漏不穩定 runtime details

Mitigation：first contract version 使用 app-neutral shapes，避免暴露 AgentSession、extension runner、provider/tool objects。

MediumSDK/extensions import migration 破壞相容

Mitigation：保留 compatibility shims；replacement path 有文件與測試後再移除。

Rollout：分期與回滾

  1. Phase 0：移除 packages/session app-specific config，改為 host-provided storage options。
  2. Phase 1：新增 owner/client types、snapshot/event/command contracts、contract tests。
  3. Phase 2：在 coding-agent 加 in-process adapter，先遷移低風險 SDK/RPC call sites。
  4. Phase 3：tests 覆蓋 prompt queue、abort、tools、compaction、switch/fork、JSONL writes 後，再搬 runtime ownership internals。
  5. Phase 4：另案提出 pi-sessiond 或 Web attach transport。

Rollback：每個 phase 都保留現有 shim imports 與 direct runtime path，直到 replacement path 通過 tests。
